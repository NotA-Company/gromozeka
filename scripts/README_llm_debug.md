# LLM Debug Query Scripts

This directory contains scripts for debugging and testing LLM interactions from JSON logs.

## Overview

The Gromozeka bot can log LLM interactions to JSON files. These scripts help you:

1. **Convert raw JSON logs to human-readable format** (script 1)
2. **Re-run queries from logs with modifications** (script 2 - this file)

## Script 2: `run_llm_debug_query.py`

Re-run LLM queries from human-readable JSON logs. Useful for iterating on system prompts, debugging responses, and testing changes without triggering the full bot.

### Usage

```bash
# Run the query as-is using the original model
./venv/bin/python3 scripts/run_llm_debug_query.py log.readable.json

# Override the model to use a different one
./venv/bin/python3 scripts/run_llm_debug_query.py log.readable.json --model gpt-4

# Use custom config directories
./venv/bin/python3 scripts/run_llm_debug_query.py log.readable.json \
    --config-dir configs/00-defaults \
    --config-dir configs/local \
    --dotenv-file .env.local
```

### Features

- **Load logs**: Reads human-readable JSON files generated by script 1
- **Reconstruct messages**: Properly rebuilds `ModelMessage` objects including tool calls
- **Model override**: Switch to a different model without editing the log file
- **Comparison mode**: Shows side-by-side comparison of original vs new response
- **Error handling**: Graceful handling of model not found, API errors, etc.

### Workflow Example

1. Enable JSON logging in your LLM config:
   ```toml
   [json-logging]
   enabled = true
   file = "llm.json"
   ```

2. Run the bot and trigger an LLM interaction

3. Convert the raw log to human-readable:
   ```bash
   # (Use script 1 - convert_llm_logs.py)
   ./venv/bin/python3 scripts/convert_llm_logs.py llm.json > debug.readable.json
   ```

4. Edit the `debug.readable.json` file:
   - Modify the system prompt
   - Add/remove messages
   - Change request parameters

5. Re-run the query:
   ```bash
   ./venv/bin/python3 scripts/run_llm_debug_query.py debug.readable.json
   ```

6. Compare the results and iterate!

### Output Format

The script prints:

1. **Metadata**: Model name, original date, original status
2. **Query context**: Message count and system prompt preview
3. **Result**: Status, error (if any), and response text
4. **Comparison** (if original response available): Side-by-side preview
5. **Details** Tokens used, tool calls, etc.

### Exit Codes

- `0`: Query executed successfully
- `1`: Error loading file, model not found, or API error

### File Format

The script expects a human-readable JSON file with the following structure:

```json
[
  {
    "date": "2026-05-10T12:34:56.789+00:00",
    "model": "deepseek/deepseek-v4-flash",
    "provider": "OpenrouterProvider",
    "raw": "...raw provider response...",
    "request": [
      {
        "role": "system",
        "content": "You are a helpful assistant..."
      },
      {
        "role": "user",
        "content": "What is 2+2?"
      }
    ],
    "response": "The answer is 4.",
    "status": "FINAL"
  }
]
```

- `model`: Model identifier (e.g., `gpt-4`, `openrouter/claude-haiku-4.5`)
- `request`: Array of message dictionaries with `role` and `content`
- `response`: The model's response text (for comparison)
- `status`: Result status (`FINAL`, `ERROR`, etc.)

### Tool Calls

The script handles tool calls automatically. If the original request included `tool_calls` or `tool_call_id`, they will be reconstructed as `LLMToolCall` objects.

### Notes

- The script loads config using the same pattern as `main.py`
- It supports both single-entry and multi-entry JSON files
- Multiple entries in a file are supported, but only the first entry is used
- Comment lines (starting with `#`) in the JSON file are ignored
