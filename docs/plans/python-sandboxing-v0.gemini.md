Отличная и глубокая проработка! Идея с базовым классом (языконезависимым) и абстракцией над Docker — это классический, архитектурно правильный подход к построению систем вроде Remote Code Execution (RCE) песочниц. 

Твоя задумка с монтированием `home` директории в `ro` (read-only) для библиотек, чтобы их переиспользовать, но не дать сломать — это очень умный ход.

Давай разберем твой дизайн, внесем корректировки, добавим недостающее и составим план.

---

### 1. Как стоит изменить текущие методы?

*   **Асинхронность.** Если твой бот написан на `aiogram`, `FastAPI` или любом другом асинхронном фреймворке, эта библиотека **обязана быть асинхронной**. Синхронный Docker SDK заблокирует Event Loop бота. Используй библиотеку `aiodocker` вместо стандартной `docker`. Следовательно, все методы должны быть `async def`.
*   **Тип `containerId`**. Лучше использовать `str` (UUID сессии пользователя или диалога), так как `int` ограничивает тебя, а Docker работает со строковыми именами и хешами.
*   **Абстракция установки зависимостей.** Базовый класс (`BaseSandbox`) не должен ничего знать про `pip` или `requiredLibs`. У него должен быть метод `async def build_environment(dependencies: List[str]) -> bool`. А уже в наследнике `PythonSandbox` этот метод будет вызывать `pip install`, а в `TypeScriptSandbox` — `npm install`.
*   **Осторожнее с `upgradeLibs`**. В парадигме песочниц обычно не обновляют библиотеки "на лету", так как это может сломать уже написанный код в рамках сессии. Лучше просто позволять доустанавливать новые через `createContainer` (или метод `update_environment`).

### 2. Какие еще методы нам нужны?

Чтобы LLM могла полноценно работать как Data Scientist или программист, ей нужно не только читать, но и писать/загружать файлы:

1.  `putFileContent(containerId: str, fileName: str, content: bytes/str) -> bool`
    Позволяет боту (или юзеру через бота) загрузить в вольюм контейнера датасет (CSV, JSON, картинку), который скрипт потом будет обрабатывать.
2.  `listFiles(containerId: str, path: str = ".") -> List[Dict]`
    Возвращает список файлов в рабочей директории вольюма (с размерами). LLM часто генерирует файлы (например, графики `.png`) и ей нужно сделать "ls", чтобы узнать, как файл назвался, чтобы потом отдать его юзеру.
3.  `getContainerStatus(containerId: str) -> str`
    Быстрая проверка: жив ли контейнер, убит ли по OOM (Out of Memory) или завис.
4.  `killScript(containerId: str) -> bool`
    Если пользователь видит, что скрипт долго выполняется, у него должна быть кнопка "Остановить" в боте.

### 3. Настройки и ограничения Docker (Security & Limits)

Твой `extraConfig` и дефолты должны обязательно транслироваться в следующие параметры Docker API, иначе песочницу взломают или она положит сервер:

*   `mem_limit="512m"` — ограничение RAM.
*   `memswap_limit="512m"` — **Обязательно!** Равно `mem_limit`, чтобы запретить контейнеру уходить в swap на жесткий диск.
*   `nano_cpus=1000000000` — ограничение CPU (1 ядро).
*   `pids_limit=64` — **Обязательно!** Защита от "Fork-бомб" (когда LLM пишет скрипт, который бесконечно плодит процессы).
*   `security_opt=["no-new-privileges"]` — запрещает процессам получать дополнительные права (например, через `sudo`).
*   `cap_drop=["ALL"]` — отбирает все Linux-привилегии (capabilities) у контейнера.
*   `network_mode="none"` (если `allowNetwork=False`) — жесткое отключение сети.
*   `read_only=True` — делает корневую файловую систему (`/`) только для чтения. На запись монтируется *только* папка `workspace` (через вольюмы).

### 4. Как должен выглядеть конфиг?

Лучше всего описать его через `pydantic` или `dataclasses`, чтобы иметь валидацию "из коробки":

```python
from pydantic import BaseModel
from typing import Optional, List

class ResourceLimits(BaseModel):
    memory_mb: int = 512
    cpus: float = 1.0
    pids: int = 64
    max_execution_time_sec: int = 300

class Timeouts(BaseModel):
    container_ttl_minutes: int = 30 # Время жизни простаивающего контейнера
    volume_ttl_minutes: int = 120   # Время жизни файлов после удаления контейнера

class SandboxConfig(BaseModel):
    base_image: str = "python:3.11-slim" # ВАЖНО: не alpine!
    volumes_dir: str = "/tmp/sandbox_data"
    limits: ResourceLimits = ResourceLimits()
    timeouts: Timeouts = Timeouts()
    default_network_disabled: bool = True
```
*(Примечание: Как мы обсуждали ранее, используй `debian-slim` (например, `python:3.11-slim`), а не `alpine`, иначе установка `numpy`/`pandas` будет каждый раз компилироваться из исходников часами).*

### 5. Что еще стоит предусмотреть (Подводные камни)

1.  **OOM (Out Of Memory) Detection:** Если скрипт съест всю память, Docker убьет его **без предупреждения**. Твой `runScript` должен проверять код возврата (Exit Code). Если он равен `137`, значит сработал OOM Killer. Метод должен возвращать в `ScriptRunResult` понятную ошибку для LLM: *"Скрипт убит из-за превышения лимита памяти"*.
2.  **Concurrency (Гонка данных):** Что будет, если LLM сгенерирует два вызова функции `runScript` одновременно для одного `containerId`? Нужна блокировка (Lock) на уровне сессии.
3.  **Изоляция пользователей:** Убедись, что права на файлы в Volume на хост-машине корректные, чтобы скрипт из Контейнера А не смог через симлинки или обход путей прочитать Volume Контейнера Б.
4.  **Пользователь без рута:** Внутри Dockerfile обязательно создай пользователя `sandbox_user` и выполняй код от него, а не от `root`.

### 6. Краткий план для обсуждения с командой (Pitch)

Вот структура, которую можно скинуть в рабочий чат или показать на созвоне:

#### 🛠 Архитектура компонента: "Agnostic Code Sandbox"

**1. Цель:**
Создать безопасную, переиспользуемую и языконезависимую библиотеку-песочницу на базе Docker для выполнения сгенерированного LLM кода (с поддержкой сохранения стейта файлов и библиотек в рамках диалога).

**2. Концепция базового слоя (Core):**
*   **Абстрактный класс `BaseSandbox`**: Управляет жизненным циклом Docker-контейнеров, монтированием Volume, лимитами ресурсов и Garbage Collection. Не привязан к конкретному языку.
*   **Имплементации (например, `PythonSandbox`)**: Наследуют базу, определяют базовый Docker-образ (не Alpine, а Debian-slim), реализуют специфичную логику установки пакетов (`pip install`) и команду запуска скриптов.

**3. Управление состоянием (State Management):**
*   **Сессии**: 1 пользовательский диалог = 1 `containerId` (изолированный Volume).
*   **Кэширование зависимостей**: Библиотеки ставятся в отдельную директорию на Volume и монтируются в Read-Only, чтобы сохранить их между запусками, но защитить от модификации скриптами.
*   **Garbage Collector**: Фоновый процесс очищает контейнеры, неактивные более `X` минут, и Volumes, неактивные более `Y` минут.

**4. Безопасность и стабильность (Security First):**
*   Жесткие Docker-лимиты: CPU, RAM + отключение Swap, ограничение PIDs (защита от fork-бомб).
*   Drop Capabilities + выполнение от non-root пользователя.
*   Опциональное отключение сети (`network=none`) во время выполнения кода.
*   Перехват OOM (Exit Code 137) и таймаутов выполнения.

**5. Интерфейс для LLM (API):**
Методы для выполнения скриптов (возвращают stdout/stderr), чтения/записи файлов на Volume (для датасетов/артефактов) и просмотра установленных библиотек. Поверх этого будет слой санитайзинга и rate-лимитов со стороны самого бота.
