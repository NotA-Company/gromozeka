## Part B — The Implementation Journey (Process Narrative)

This section tells the story of how the feature was built, in order, including where plans met reality and where they diverged.

### Phase 0 — Architect Validation (Before Any Code)

Before dispatching any implementation work, an architect pass was run against the 892-line plan (`docs/plans/divination-handler-plan.md`) to validate every code reference against the current source tree. This single pass caught six plan-vs-reality mismatches and established an authoritative corrected brief for all downstream implementers.

**Mismatches caught by the architect:**

| Plan reference | Reality | Impact if uncaught |
|---|---|---|
| `CommandCategory.AI` (plan §5.3) | Does not exist; only `CommandCategory.TOOLS` | Handler would fail to import |
| `MessageSender.displayName` (plan §5.2b) | Attribute is `.name` | `AttributeError` at runtime |
| `MediaStatus` / `MessageType` used as strings | They are enums | Type errors, wrong comparisons |
| `LLMService.registerTool` with nested signature | Flat signature: `(name, description, parameters, handler)` | Tool registration silently broken |
| Sync `cursor` migration pattern in `docs/llm/database.md` | Reality: `async def up(self, sqlProvider: BaseSQLProvider)` with `ParametrizedQuery` | Migration would not run |
| `docs/llm/handlers.md` listed `CommandCategory.AI` and `INFO` | Neither exists in codebase | Misleading docs; the correction was deferred to a follow-up ticket |

The corrected brief became the sole input to all implementation slices. Zero guesswork during coding.

- No bot deps, no DB deps. Boundary enforced by `test_imports.py`.

**Slice 2 — DB migration 014 + `DivinationsRepository`**

- `DivinationsRepository.insertReading(...)` with keyword-only arguments and best-effort (log-and-swallow) semantics.
- 4 tests in `internal/database/repositories/test_divinations.py` (schema + insert + error path).
- Minor note: the test-database fixture behavior differed slightly from what `docs/llm/database.md` implied; the slice handled it locally rather than triggering an upstream doc fix at that moment.

**Slice 3 — `ChatSettingsKey` entries + config defaults**
- 5 new keys added to `internal/bot/models/chat_settings.py`.
- Default prompt strings (system prompts, user template, image template, reply template) added under `[bot.defaults]` in `configs/00-defaults/bot-defaults.toml`.
- `configs/00-defaults/divination.toml` created with feature-flag structure.
- 13 tests in `tests/integration/test_divination_chat_settings.py`.

After all three slices merged (serially in code, conceptually parallel), the **handler slice** ran:

Golden scaffolding, verifier sweep, and docs were added last.

### Phase 2 — Boundary Test Pollution Fix

Slice 1's `test_imports.py` was initially written using `sys.modules` membership to check that `lib.divination` does not pull in `internal.services.llm` or `internal.bot`. This approach has a critical flaw: `sys.modules` is process-global, and other tests in the same pytest session may have already imported those modules, making the check meaningless (it would never catch a real leak) or produce false positives (it would flag modules loaded by unrelated tests).

**Fix:** The test spawns a fresh interpreter via `subprocess.run([sys.executable, "-c", probeCode], ...)`, imports `lib.divination` there, and JSON-dumps the list of forbidden-prefix modules back over stdout. This is fully isolated from the pytest session. See [`lib/divination/test_imports.py`](../../lib/divination/test_imports.py).

```python
probeCode: str = (
    "import importlib, json, sys;"
    " importlib.import_module('lib.divination');"
    " forbiddenPrefixes = ('internal.services.llm', 'internal.bot');"
    " leaked = sorted(m for m in sys.modules if m.startswith(forbiddenPrefixes));"
    " print(json.dumps(leaked))"
)
result = subprocess.run([sys.executable, "-c", probeCode], capture_output=True, text=True, check=True)
assert result.stdout.strip() == "[]"
```

This pattern is now the canonical approach for any "module X must not import Y" test in this repo.

### Phase 3 — User Refinement Round 1

After the initial handler landed, the user reviewed and requested seven changes to `_handleReading`:

1. **Take resolved `layout` + `question`** instead of raw `args` — the parse/validate layer was extracted into `_handleReadingFromArgs` (slash-command path) and `_runReadingForTool` (tool path), so `_handleReading` always receives clean, validated inputs.
2. **LLM-tool path: no text reply to user** — only image (if generated) is sent as a bot message; the LLM gets interpretation via JSON tool result.
3. **Bump `TypingAction` to `UPLOAD_PHOTO`** before image generation starts, mirroring `MediaHandler`'s pattern.
4. **Drop `media_attachments` insert** + `media_id` column — the handler no longer inserts into `media_attachments` before calling `sendMessage`; `sendMessage` handles the media pipeline internally. The `media_id` column was also dropped from the migration.
5. **No caption truncation** — removed the manual `[:CAPTION_LIMIT]` slice; `sendMessage` handles overflow internally.
6. **Drop `rng_seed` column** — the `divinations` table was simplified; no seed persisted in v1.
7. **Pass `imagePrompt` to `sendMessage` as `mediaPrompt=`** — for consistent media metadata handling.

**Edit-in-place migration:** Because `divination.enabled = false` by default and no deployment had applied the v1 schema, migration 014 was edited in place (dropping `media_id` and `rng_seed` columns) rather than stacking a corrective migration 015. This is a deliberate choice with a clear criterion: **feature flag still off + no production rollout + still pre-merge → edit in place**.

### Phase 4 — Structured Reply Template

The user asked for a structured user-visible reply on the slash-command path that would show the drawn cards/runes explicitly, so users can verify the LLM did not hallucinate symbols it didn't receive.

Changes:

- **`BaseDivinationSystem.renderReplyTemplate(template, *, layoutName, drawnSymbolsBlock, interpretation)`** — applies `_safeFormat` to the template.

- New `lib/divination/test_base_render_block.py` (289 lines) covers all rendering branches.

### Phase 5 — Verifier Catches a Real Production Bug

During the structured-template round, a staging issue was introduced in the working tree: the handler file in the git index (staged version) contained a **duplicate `sendMessage` call** — the result of a rebasing/editing artifact. The working tree was correct, but if the user had run `git commit`, the staged version would have shipped with two bot messages sent per reading on the slash-command path.

The verifier caught this by running `git status --short` and comparing the staged diff vs. working-tree diff. A follow-up agent staged the working tree version and added two regression tests:

- `test_taroCommandSendMessageCalledOnce` — asserts `sendMessage.call_count == 2` (image + text, two separate calls) on the slash-command path.
- `test_runesCommandSendMessageCalledOnce` — same for the runes path.

**Lesson:** Verifiers must check `git status --short` and inspect both the index and working tree, not just run `make test` (which always tests the working tree, not the staged version).

A second issue surfaced in the same round: the implementer's report claimed they "fixed pre-existing test failures around `drawsJson`." The verifier ran `git show HEAD:<file>` to inspect the pre-change state and confirmed nothing was actually broken before — the framing was inaccurate. **Lesson:** Any agent claim about "fixed pre-existing failures" must be independently ground-truthed by the verifier with `git show HEAD:` and `git diff HEAD`.

### Phase 6 — User-Driven Refactor: Split Send

The user manually revised the send logic on the slash-command path:

- **Before:** image + interpretation text sent as a single `sendMessage(photoData=..., messageText=...)` call (combined photo-with-caption).
- **After:** two separate `sendMessage` calls — first one sends the image (no text), second sends the text reply (no image). This gives cleaner display in clients and allows the text to exceed the Telegram caption limit without special handling.

Additionally, the LLM-tool JSON result was trimmed: `summary` and `imageGenerated` keys were dropped. The host LLM doesn't need a summary; `done` + `layout` + `draws` + `interpretation` are sufficient for the orchestrating model to compose its own reply.

These changes broke existing tests. A test-fixer agent updated assertions per a three-category taxonomy: **Change A** (image message no longer has caption), **Change B** (text message is now a separate call), **Change C** (tool JSON shape change). No production-code changes from the team in this round.

### Phase 7 — Rune Unicode Glyphs

The user requested that each Elder Futhark rune carry its Unicode glyph character so user-facing replies can display a visual lead-in (e.g. `ᚠ Fehu` → `ᚠ Феху`).

Changes:
- **`Symbol.glyph: Optional[str] = None`** added to [`lib/divination/base.py`](../../lib/divination/base.py:56).
- All 24 Elder Futhark runes in [`lib/divination/decks/runes_elder_futhark.py`](../../lib/divination/decks/runes_elder_futhark.py) populated with Unicode literals (ᚠ ᚢ ᚦ ᚨ ᚱ ᚲ ᚷ ᚹ ᚺ ᚾ ᛁ ᛃ ᛇ ᛈ ᛉ ᛊ ᛏ ᛒ ᛖ ᛗ ᛚ ᛜ ᛞ ᛟ), **not** `\uXXXX` escape sequences. Literals are more readable in diff and make encoding errors visible immediately.
- Tarot cards retain `glyph=None` — there is no single canonical Unicode glyph for the RWS deck.
- `BaseDivinationSystem.renderDrawnSymbolsBlock` prepends `f"{draw.symbol.glyph} "` when glyph is present.
- Tool JSON gains `"glyph"` key in the `draws` list.
- `lib/divination/decks/test_decks.py` extended with 31 lines of glyph-coverage tests.
- `lib/divination/test_base_render_block.py` extended with 53 lines testing glyph rendering and no-glyph tarot path.
- `tests/bot/test_divination_handler.py` extended with 69 lines of regression tests.

This phase added 32 tests (1646 → 1678).

## Part C — Insights and Patterns Worth Re-Using

The most durable value in this document: process patterns that generalise across features.

### 1. The Architect-First Pattern

**Spending one agent invocation up-front to validate the plan against current source code prevented six implementation errors before a single line was written.** The plan was 892 lines; the validation pass took less time than one failed test run would have cost.

**Rule:** Always run an architect validation pass before parallel implementation when the plan is >100 lines. The architect's output is a corrected brief, not a design doc — it lists what's wrong in the plan and what the correct code reference is.

### 2. Slice Parallelism Rules

Three slices ran in parallel (`lib/divination`, DB migration + repo, `ChatSettingsKey` + config) because they touched entirely disjoint file sets. The handler slice depended on all three and ran serially after.

**Pattern:** Build a file-set DAG before dispatching agents. Parallelise only the leaves — slices that share files will conflict and create merge confusion. For this feature, the DAG was:

```
lib/divination  ──┐
DB + repo       ──┤──> DivinationHandler ──> golden + docs
ChatSettings    ──┘
```

### 3. Verifier as a Hard Gate, Not a Rubber Stamp

Two real production issues (the boundary-test pollution, the staged-vs-working-tree bug) and one misleading agent self-report were caught only because the verifier had explicit checklist items:
- `git status --short` — check both index and working tree
- `git show HEAD:<file>` — verify "pre-existing" claims
- Multi-ordering test runs (run tests in different orders to detect state leakage)

**Never skip the verifier on multi-slice features.** The cost of a verifier pass is one agent invocation; the cost of a staged production bug is a hotfix.

### 4. Edit-In-Place Migrations

When a feature ships disabled by default and no deployment has applied the v1 schema, editing the migration in place beats stacking a corrective migration.

**Criterion for edit-in-place:**
- Feature flag is still `false` in all defaults → no user has hit the handler
- No production deployment has run the migration
- Still pre-merge (the migration hasn't left the feature branch)

If any condition fails, add migration N+1 instead. This feature satisfied all three.

### 5. Boundary Tests via `subprocess`

`sys.modules` checks in pytest are unreliable: other tests in the session may have already imported the forbidden module. The correct approach is a subprocess probe:

```python
probeCode: str = (
    "import importlib, json, sys;"
    " importlib.import_module('lib.your_library');"
    " forbiddenPrefixes = ('internal.services.llm', 'internal.bot');"
    " leaked = sorted(m for m in sys.modules if m.startswith(forbiddenPrefixes));"
    " print(json.dumps(leaked))"
)
result = subprocess.run([sys.executable, "-c", probeCode], capture_output=True, text=True, check=True)
assert result.stdout.strip() == "[]"
```

**Apply this pattern to any `lib/*` package that must stay decoupled from `internal/`.**

### 6. Stale Docs Are Real Liabilities

Three doc references were stale at the start of this feature (`CommandCategory.AI`, sync-cursor migration pattern in `docs/llm/database.md`, `MessageSender.displayName`). They cost zero implementation time **only because the architect caught them before any code was written.** Without that pass, each would have caused a test-run failure, a debugging session, and a back-and-forth round trip.

**Recommendation:** When an architect pass flags stale doc references, open a cleanup ticket immediately. Don't defer "doc debt" to the same round as the feature work — but don't block the feature on it either.

### 7. Agent Self-Reports Must Be Ground-Truthed

Two implementers in this run made claims that were misleading or false:
- An implementer claimed they "fixed pre-existing test failures around `drawsJson`." The verifier's `git show HEAD:` confirmed nothing was broken before the change.
- An implementer submitted a file where the staged (index) version had a bug the working tree didn't.

**Process rule:** Any agent claim about "fixed pre-existing X" or "no changes to Y" must be ground-truthed by an independent verifier using `git show HEAD:<path>` and `git diff HEAD`. Trust the diff, not the narrative.

### 8. The Russian-Localisation Pattern

English is the source-of-truth in `lib/divination`; Russian translations live in `localization.py` as plain dicts keyed by English string. Every `Symbol.name`, layout name, and position name has an entry. The `tr(table, enKey, lang)` helper returns the translation or falls back to the English original.

- Adding a new language is **data-only**: extend the existing dicts in `localization.py`.
- Adding a new symbol/layout/position requires both the English entry (in the deck or layout file) **and** a `localization.py` entry — enforced by `test_localization.py`.
- Unicode glyphs (rune characters) **bypass localisation entirely** — they are universal. The `Symbol.glyph` field is emitted as-is in all locales.

**Pattern:** Keep English source-of-truth, translate at the leaf (localisation module), bypass for truly universal data (numbers, glyphs, currency codes).

### 9. Composite Natural PK > `AUTOINCREMENT`

`divinations` uses `PRIMARY KEY (chat_id, message_id)` — the originating user message. This pattern:
- Works identically on SQLite, PostgreSQL, and MySQL.
- Provides a natural uniqueness constraint (you can't draw the same reading twice for the same message).
- Requires no dialect-specific DDL.

**Project rule confirmed:** never use `AUTOINCREMENT` / `SERIAL` / `AUTO_INCREMENT`. Pick a composite natural key when the row is identifiable by external IDs the app already holds.

### 10. Tool-Path JSON Design

Returning structured data (`layout`, `draws`, `interpretation`) for the host LLM to consume is more robust than returning a free-text summary. The host LLM can compose its own user-facing reply; the bot only needs to send the image. The initial plan included `summary` and `imageGenerated` keys; both were dropped in Phase 6 because they were redundant — the host LLM can derive a summary from `interpretation` and doesn't need to know whether the bot sent an image.

**Pattern:** **Let the LLM compose its own user-facing text from structured tool returns whenever the LLM is the orchestrator.** Avoid baking text summaries into tool results — they constrain the LLM's phrasing and duplicate information.

Final tool JSON shape on success:
```json
{
  "done": true,
  "layout": {"id": "three_card", "en": "Three-Card Spread", "ru": "...", "positions": [...]},
  "draws": [{"nameEn": "The Fool", "nameRu": "Шут", "glyph": null, "reversed": false, "position": "Past", "meaning": "..."}],
  "interpretation": "<LLM text>"
}
```

### 11. Golden-Test Trade-Off

The aurumentation scaffolding was written without live LLM fixtures because recording requires a real API key. The replayer test in `test_golden.py` falls back to a sanity-only mode (validates scenario file structure and layout resolution) when `data/` is empty. Five fixture JSON skeletons (metadata + placeholder `responses` fields) were committed in Phase 7 to ensure the collector knows which scenarios to record.

**Pattern:** Ship the recorder (`collect.py`) + scenario definitions (`input/scenarios.json`) + fixture skeletons in the PR. Ask the operator to run `collect.py` and commit populated fixtures separately. Document it as a known limitation. The suite stays green on CI without any API credentials.

**Caveat for this feature:** Byte-equality checking on `buildInterpretationMessages` output is not yet active because `lib.aurumentation.collectGoldenData` does not propagate runner return values into fixture metadata (see Part D, item 1).

## Part D — Known Limitations / Follow-Up Tickets

These are actionable items that are out of scope for this PR but should be tracked.

### D-1. Aurumentation Prompt-Rendering Byte-Equality Not Active

**What:** `test_golden.py` can replay recorded HTTP traffic and confirm the LLM returned the same response. It cannot yet detect a regression in `buildInterpretationMessages` output (prompt text) because `lib.aurumentation.collectGoldenData` does not propagate runner return values into fixture metadata — so the expected messages are not stored in the fixture file.

**Impact:** A change to `cardsBlock` formatting or `localization.py` would not be caught by the golden replayer until a human manually compares prompts.

**Fix options:**
1. Extend `lib.aurumentation` collector with an `extraMetadata` callback that the runner calls with the rendered messages.
2. Add a post-collection script that re-runs scenarios deterministically (pinned RNG seed) and writes `expected_messages` into each fixture.

Either is a ~1-day ticket.

### D-2. `docs/llm/database.md` Shows Stale Migration Pattern

**What:** The migration examples in [`docs/llm/database.md`](../llm/database.md) still show the old sync pattern:

```python
# Stale — do not copy
def up(self, cursor):
    cursor.execute(...)
```

The current reality (since at least migration 010) is:

```python
# Correct
async def up(self, sqlProvider: BaseSQLProvider) -> None:
    await sqlProvider.batchExecute([ParametrizedQuery("...")])
```

The same doc also contains `AUTOINCREMENT` examples that contradict AGENTS.md. Worth a focused doc-cleanup ticket (estimated: 2–3 hours).

### D-3. `docs/llm/handlers.md` Lists Non-Existent `CommandCategory` Values

**What:** As of the architect pass, the doc listed `CommandCategory.AI` and `CommandCategory.INFO`. Neither exists in [`internal/bot/models/__init__.py`](../../internal/bot/models/__init__.py). Only `CommandCategory.TOOLS` and `CommandCategory.SYSTEM` (and similar) exist. Should be fixed in the same doc-cleanup ticket as D-2.

### D-4. `media_id` Foreign-Key Enforcement Is Comment-Only

**What:** Several tables (including `chat_messages`) reference `media_attachments.file_unique_id` via comment-only "Foreign key to …" rather than a declared `FOREIGN KEY` clause. This was intentional (SQLite doesn't enforce FK constraints without `PRAGMA foreign_keys = ON`), but the intent is not documented.

**Recommendation:** Either add real `FOREIGN KEY ... ON DELETE SET NULL` clauses (with `PRAGMA foreign_keys = ON` in the provider init) or add an explicit ADR comment to the migration template explaining why soft FKs are the project standard. This is out of divination scope but was flagged during the schema review.

### D-5. Caption-Limit Constant Per Platform

**What:** Telegram's photo caption limit is 1024 characters. The Max Messenger limit may differ. Currently `sendMessage` handles overflow internally with its own fallback logic, but the constant is not documented or exposed per-provider.

**Recommendation:** Confirm Max Messenger's caption limit and either document it alongside Telegram's or expose it via the bot-provider abstraction so the handler doesn't need platform-specific conditionals.

### D-6. Operator Action Item: Record Golden Fixtures

**What:** The five fixture JSON files in `tests/divination/golden/data/` contain only metadata shells; the `responses` arrays are empty placeholders. Until a real collection run is completed, the replayer test is a `[NOTSET] SKIPPED` / sanity-only run.

**How to record:**
```bash
export OPENROUTER_API_KEY=<your-key>
./venv/bin/python3 tests/divination/golden/collect.py
# Review the populated JSON files in tests/divination/golden/data/
git add tests/divination/golden/data/*.json
git commit -m "feat(divination): record golden LLM fixtures"
```

After that, the replayer asserts byte-identical HTTP traffic on every CI run.
